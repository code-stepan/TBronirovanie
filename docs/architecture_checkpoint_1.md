# Описание архитектуры (Контрольная точка 1)

## Слои

Проект разделён на 3 логических слоя:

1. Domain (`src/domain`)
   - Бизнес-сущности: `User`, `Event`, `Ticket`, `Booking`
   - Бизнес-сервис: `BookingService`
   - Правила предметной области: доступность мест, переходы статусов бронирования, верификация пользователя
   - Паттерны проектирования:
     - Builder: `UserBuilder`
     - Factory Method: `TicketFactory`

2. Interface (`src/interface`)
   - Зарезервировано для контрольной точки 2
   - В будущем должно взаимодействовать с Domain через фасад/адаптеры

3. Infrastructure (`src/infrastructure`)
   - Сохранение данных: `JSONRepository`
   - Технические детали изолированы от кода предметной области

## Основной поток предметной области

1. Создание пользователя через `UserBuilder` с обязательными полями (`first_name`, `last_name`) и необязательными полями (`passport`, `address`).
2. Создание события с вместимостью и свободными местами.
3. Создание билета через `TicketFactory` одного из поддерживаемых типов.
4. `BookingService.create_booking()` проверяет верификацию пользователя, резервирует одно место, помечает билет как забронированный и возвращает `Booking`.
5. Слой Infrastructure сохраняет данные бронирования в JSON.

## Почему выбран такой дизайн

- Domain остаётся независимым от UI и деталей хранения.
- Сложность создания объектов инкапсулирована в специализированных паттернах (Builder/Factory).
- Интерфейс Repository позволяет заменить JSON на SQL в последующих контрольных точках с минимальными изменениями.
