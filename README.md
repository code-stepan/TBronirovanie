| Паттерн | Файл | Обоснование |
|---------|------|-------------|
| Observer | domain/services.py | Система уведомлений при изменении статуса брони (Email, SMS). Слабая связанность события и подписчиков. |
| State | domain/models.py | Управление статусом брони (Created → Paid → Cancelled → Refunded). Поведение зависит от состояния. |
| Command | domain/services.py | Реализация отмены транзакции (Undo). Каждая операция инкапсулирована в объект-команду. |

## ✅ Принципы SOLID

| Принцип | Реализация в проекте |
|---------|---------------------|
| Single Responsibility | Каждый класс решает одну задачу (например, Ticket не сохраняет себя в БД) |
| Open/Closed | Система открыта для расширения (новый тип билета через Factory), закрыта для модификации |
| Liskov Substitution | ConsoleUI и GUIUI взаимозаменяемы без поломки логики |
| Interface Segregation | Узкие интерфейсы (BaseUI с 5-6 методами, не больше) |
| Dependency Inversion | Зависимость от абстракций (Domain зависит от интерфейса репозитория) |

## 📋 Git Workflow

| Ветка | Назначение |
|-------|------------|
| main | Стабильная версия (сливает куратор) |
| dev | Основная разработка |
| checkpoint_1, checkpoint_2, checkpoint_3 | Ветки для сдачи чекпоинтов |

Процесс сдачи:
1. Создать ветку checkpoint_x из dev
2. Создать Pull Request из checkpoint_x в main
3. Добавить ревьюера (куратора)
4. При запросе правок — вносить в checkpoint_x, мержить в dev
5. В main сливать только после аппрува куратора
